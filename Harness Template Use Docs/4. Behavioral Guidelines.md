# Behavioral Guidelines

이 문서는 Harness Template을 실제 프로젝트에 적용할 때 사람이 맡아야 할 일을 정리합니다.

AI는 빠르게 구현할 수 있지만, 제품 방향과 위험 판단은 사람이 결정해야 합니다. 사람의 역할은 코딩보다 판단, 승인, 우선순위 조정에 가깝습니다.

## 1. PRD 작성

사람이 먼저 정해야 하는 항목입니다.

- 제품 목적
- 핵심 사용자
- 핵심 플로우
- MVP 범위
- 하지 않을 것
- 성공 기준

PRD가 모호하면 AI는 그 모호함을 코드로 확정해버릴 수 있습니다. 구현 전에 제품 판단 기준을 먼저 적어두는 것이 중요합니다.

## 2. 모호한 요구사항 결정

AI가 `open question`으로 남긴 것 중 제품 방향에 영향을 주는 항목은 사람이 결정합니다.

예시:

- 권한 정책
- 무료/유료 범위
- 데이터 보존 기간
- MVP 제외 범위
- 사용자에게 보여줄 오류 메시지 수준
- 외부 서비스 장애 시 대응 방식

## 3. 테스트 명령 확인

`scripts/init-testing-commands.ps1`가 감지한 명령이 실제 프로젝트에 맞는지 사람이 확인합니다.

잘못 감지됐거나 감지하지 못한 경우에는 `docs/TESTING.md`를 직접 수정합니다.

확인할 명령:

- install
- dev
- test
- lint
- typecheck
- build

## 4. 작업 우선순위 승인

`docs/exec-plans/active/`에 쪼개진 작업 중 무엇부터 할지 사람이 고릅니다.

AI에게 모든 작업을 한 번에 시키면 큰 파일, 섞인 책임, 검증 누락이 다시 생길 수 있습니다. 반드시 하나의 기능 단위로 진행합니다.

## 5. UI/UX 판단

AI가 만든 화면이 동작하더라도 실제 사용성이 괜찮은지는 사람이 확인해야 합니다.

확인할 항목:

- 핵심 플로우가 자연스러운지
- 모바일/데스크톱에서 깨지지 않는지
- 로딩/빈/오류 상태가 있는지
- 버튼, 입력, 상태 메시지가 이해하기 쉬운지
- 불필요한 설명 문구나 임시 UI가 남아 있지 않은지

## 6. 보안/비용/배포 결정

아래 항목은 사람이 승인해야 합니다.

- API 키와 시크릿 위치
- 결제 또는 유료 기능 범위
- LLM/API 비용 제한
- 배포 환경
- 도메인과 인증 설정
- 롤백 기준
- 사용자 데이터 삭제/보존 정책

## 7. 커밋/PR 승인

AI가 커밋 메시지와 PR 본문을 제안하게 하고, 사람이 최종 승인합니다.

확인할 항목:

- 한 커밋이 하나의 목적만 갖는지
- PR이 하나의 목적만 갖는지
- 테스트/검증 결과가 적혀 있는지
- 민감정보가 포함되지 않았는지
- 기능 변경과 리팩터링이 섞이지 않았는지

## Core Principle

```text
PRD → 하네스 변환 → 작은 실행 계획 → 기능 하나 구현 → 검증 → 커밋/PR
```

이 순서를 지키면 큰 파일에 모든 코드를 몰아넣는 문제를 줄이고, AI가 만든 변경을 사람이 판단 가능한 크기로 유지할 수 있습니다.
