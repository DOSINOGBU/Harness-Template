# Running Prompt

아래 프롬프트는 프로젝트용 하네스 초기화가 끝난 뒤, 실제 구현을 시작할 때 사용합니다.

한 번에 전체 앱을 만들게 하지 않고, `docs/exec-plans/active/`에 있는 작업 중 하나만 구현하게 하는 것이 핵심입니다.

```text
이제 `docs/exec-plans/active/`에 있는 MVP 작업 중 가장 우선순위가 높은 것부터 하나만 구현해줘.

규칙:
- 작업 시작 전에 `scripts/bootstrap-agent-context.ps1`를 실행해서 시작 컨텍스트를 확인해줘.
- 작업 전 관련 문서를 읽어줘.
- 한 번에 하나의 기능만 구현해줘.
- 큰 파일이 생기지 않게 책임을 분리해줘.
- 로딩/빈/오류 상태를 포함해줘.
- 선택한 계획이 downstream 기능의 입력 품질을 결정하는 경우, 단순 구현 완료만으로 completed 처리하지 말고 품질 검증 또는 후속 repair 계획을 남겨줘.
- 구현 후 테스트/검증을 실행하고 결과를 보고해줘.
- fake/mock 테스트 통과와 실제 샘플 품질 통과를 구분해서 보고해줘.
- 핵심 산출물이 사람이 승인해야 하는 결과물이라면, 승인 전에는 카드/퀴즈/UI 같은 후속 기능의 품질 완료로 간주하지 마.
- 완료 전에 `.harness/checklists/pre-completion.md`로 원래 요청과 검증 결과를 다시 확인해줘.
- 커밋은 아직 하지 말고 변경 요약과 검증 결과만 알려줘.
```

## 반복 사용 방식

1. AI가 구현할 작업 단위를 하나 고릅니다.
2. 사람이 작업 범위가 적절한지 확인합니다.
3. AI가 구현하고 검증합니다.
4. AI가 완료 전 self-verify를 수행하고 필요한 경우 `docs/agent-runs/`에 실행 기록을 남깁니다.
5. 사람이 동작, UX, 범위 초과 여부를 확인합니다.
6. 문제가 없으면 커밋/PR 단계로 넘어갑니다.

## 추가로 붙이면 좋은 조건

필요한 경우 아래 조건을 프롬프트 끝에 추가합니다.

```text
추가 조건:
- 새 파일이 필요하면 책임 단위별로 분리해줘.
- 500줄 이상 파일에 새 기능을 추가해야 한다면 먼저 분리 계획을 보고해줘.
- 외부 API 호출에는 실패 로그와 사용자에게 보여줄 오류 상태를 포함해줘.
- 테스트가 불가능한 부분은 수동 검증 방법을 명확히 적어줘.
- 같은 검증이 반복 실패하면 기존 접근을 반복하지 말고 원인 가설을 다시 세워줘.
- 반복 실패나 중요한 결정이 있으면 `docs/agent-runs/`에 run log를 남겨줘.
```
