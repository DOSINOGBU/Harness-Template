# First Prompt

아래 프롬프트는 PRD를 작성한 뒤, 범용 Harness Template을 프로젝트용 하네스로 초기화할 때 사용합니다.

중요한 점은 바로 코드를 구현하지 않고, 먼저 PRD를 하네스 문서로 분배하게 하는 것입니다.

```text
이 저장소는 범용 Harness Template을 기반으로 새 프로젝트용 하네스를 만드는 단계야.

먼저 코드를 구현하지 말고, 아래 순서로 PRD를 프로젝트 하네스 문서에 반영해줘.

1. `AGENTS.md`, `docs/README.md`, `docs/WORKFLOW.md`를 읽고 현재 하네스 구조를 파악해줘.
2. `docs/PRD.md`를 제품 요구사항 원본으로 읽어줘.
3. PRD 내용을 아래 문서에 역할별로 분배해줘.
   - `docs/PRODUCT_CONTEXT.md`
   - `ARCHITECTURE.md`
   - `docs/PROJECT_RULES.md`
   - `docs/DATA.md`
   - `docs/SECURITY.md`
   - `docs/FRONTEND.md`
   - `docs/BACKEND.md`
   - `docs/TESTING.md`
   - `docs/OBSERVABILITY.md`
   - `docs/RELIABILITY.md`
   - 필요하면 `docs/DEPLOYMENT.md`, `docs/COST.md`, `docs/RUNBOOK.md`

4. PRD에서 모호한 부분, 위험한 결정, 나중에 확인해야 할 질문을 정리해줘.
5. MVP 구현을 위한 작업 단위를 `docs/exec-plans/active/` 아래에 작은 계획 문서로 나눠줘.
6. `scripts/init-testing-commands.ps1`를 dry-run으로 실행해서 테스트 명령 감지 결과를 보고해줘.
7. 마지막에 다음을 보고해줘.
   - 반영한 문서
   - PRD에서 하네스로 옮긴 핵심 결정
   - 아직 사람이 결정해야 할 질문
   - 바로 구현해도 되는 작업 단위
   - 실행한 검증과 남은 warning

중요:
- 아직 기능 구현은 하지 마.
- PRD 내용을 문서에 그대로 복붙하지 말고, 각 문서 목적에 맞게 요약/분배해줘.
- 모호한 요구사항은 임의로 확정하지 말고 assumption 또는 open question으로 남겨줘.
```

## 사용 타이밍

- `docs/PRD.md` 작성 직후
- 새 프로젝트에 Harness Template을 처음 적용할 때
- 구현 전에 문서, 계획, 검증 기준을 먼저 정리하고 싶을 때

## 사람이 확인할 것

- AI가 PRD를 잘못 해석한 부분이 없는지 확인합니다.
- `open question`으로 남긴 항목 중 제품 방향에 중요한 것을 먼저 결정합니다.
- `docs/exec-plans/active/`에 생성된 작업 단위가 너무 크면 더 쪼개도록 요청합니다.
