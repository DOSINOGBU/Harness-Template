# Agent Behavior

AI 에이전트가 코드를 작성하거나 수정할 때 기본으로 지킬 행동 기준입니다.
상세한 프로젝트 규칙은 각 도메인 문서를 따르고, 이 문서는 과잉 구현과 추측 수정을 막는 짧은 기준으로 사용합니다.

## Think Before Coding

- 확실하지 않은 요구사항은 조용히 가정하지 않습니다.
- 해석이 여러 가지면 가능한 해석과 tradeoff를 먼저 드러냅니다.
- 더 단순한 접근이 있으면 구현 전에 말합니다.
- 혼란이 남아 있으면 이름 붙여 설명하고 확인합니다.

## Simplicity First

- 요청받지 않은 기능을 추가하지 않습니다.
- 한 번만 쓰는 코드에 추상화를 만들지 않습니다.
- 미래 확장성을 이유로 복잡도를 미리 넣지 않습니다.
- 확인되지 않은 시나리오를 위한 에러 처리나 설정을 과하게 만들지 않습니다.

## Surgical Changes

- 필요한 파일과 라인만 변경합니다.
- 주변 코드, 주석, 포맷을 함께 개선하지 않습니다.
- 기존 스타일이 마음에 들지 않아도 현재 작업에서는 맞춥니다.
- 내가 만든 미사용 코드만 정리하고, 기존 dead code는 발견사항으로 남깁니다.
- 모든 변경 라인은 사용자 요청이나 검증 필요성과 연결되어야 합니다.

## Goal-Driven Execution

- 작업을 검증 가능한 성공 기준으로 바꿉니다.
- 버그 수정은 가능하면 실패를 먼저 재현한 뒤 고칩니다.
- 리팩터링은 변경 전후 동작이 같음을 확인합니다.
- 여러 단계 작업은 각 단계마다 확인 방법을 둡니다.

## Self-Verification Gate

- 완료 보고 전에는 `.harness/checklists/pre-completion.md`를 기준으로 요구사항, 변경 파일, 검증 결과를 다시 확인합니다.
- 검증을 수행하지 않은 채로 완료를 선언하지 않습니다.
- 관련 검증을 실행하지 못했다면 완료처럼 말하지 않고 이유와 대체 확인을 남깁니다.
- 검증 결과는 내가 작성한 코드가 아니라 원래 요청과 사용자 성공 기준에 맞춰 판단합니다.
- 실패·부분 성공을 숨기거나 성공처럼 요약하지 않고, 원인·재현 방법·검증 결과를 남깁니다.
- 실패 로그가 있었으면 원본 맥락을 보존하고, 마지막 에러 줄만 보고 단정하지 않습니다.
- 완료 보고 섹션 순서와 세부 항목은 `AGENTS.md`의 Completion Standard와 pre-completion 체크리스트의 표준 완료 보고 항목(요청 확인, 변경 사항, 검증, 결과 확인, CodeHealth, 리스크와 다음 판단)을 따릅니다.

## Loop Recovery

- 같은 파일을 세 번 이상 고치면 요구사항 해석, 파일 책임, 테스트 신호를 다시 확인합니다.
- 같은 검증이 두 번 이상 같은 이유로 실패하면 기존 접근을 반복하지 않고 원인 가설을 다시 세웁니다.
- 새 정보 없이 같은 명령을 반복 실행하지 않습니다.
- 반복 실패가 하네스 문제라면 run log에 남기고 체크리스트나 자동화 후보로 승격합니다.

## Reasoning Budget

- 작은 문서·카피·설정 변경은 짧게 계획하고 바로 검증합니다.
- 버그 수정은 원인 확인 전 구현보다 재현과 근거 확보에 더 많은 시간을 씁니다.
- 큰 설계나 경계 변경은 계획과 검증 전략에 더 깊게 사고하고, 구현은 작은 단위로 나눕니다.
- 구현이 끝난 뒤에는 추가 기능을 떠올리기보다 검증과 리스크 확인으로 전환합니다.

## Learning Loop

- 한 번 발생한 실수는 `docs/LEARNING_LOG.md`에 메모합니다.
- 두 번 반복되면 관련 checklist에 반영합니다.
- 세 번 반복되면 `docs/AUTOMATION_EVOLUTION.md`에 자동화 후보로 올립니다.
