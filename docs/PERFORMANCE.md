# Performance

성능을 미리 최적화하지 않되, 명백한 함정은 피합니다.
사용자가 체감하는 지점부터 측정 후 다룹니다.

## Default Budgets

| 영역 | 기준 | 비고 |
|---|---|---|
| 페이지 첫 응답 | 1초 이하 | 외부 의존 다수면 별도 합의 |
| API 평균 응답 | 300ms 이하 | 데이터 양이 큰 엔드포인트는 별도 |
| 백그라운드 작업 | 합의된 SLA 내 | 큐 지연을 함께 고려 |
| 빌드 시간 | 개발 흐름을 끊지 않는 수준 | CI 캐시 활용 |

> 프로젝트 특성에 맞게 위 기준을 조정합니다. 합의되지 않은 기준은 추측하지 않습니다.

## Common Pitfalls

| 패턴 | 문제 | 대응 |
|---|---|---|
| N+1 쿼리 | DB 왕복이 폭증 | 일괄 조회, 조인, 캐시 |
| 큰 응답 전체 직렬화 | 메모리/지연 증가 | 페이징, 필드 선택 |
| 동기 외부 호출 직렬 처리 | 총 지연 누적 | 병렬화 또는 배치 |
| 무제한 재시도 | 장애 증폭 | 횟수, 백오프, 회로 차단 |
| 무한 캐시 | 오래된 데이터 노출 | TTL과 무효화 규칙 |

## Measurement First

- 추측으로 최적화하지 않습니다. 측정 결과를 먼저 적습니다.
- 측정값에는 환경, 데이터 크기, 반복 횟수를 함께 기록합니다.
- 최적화 전후 수치를 비교 가능한 형태로 남깁니다.

## When To Stop

- 사용자가 체감하는 한계 안에 들어왔으면 멈춥니다.
- 가독성과 유지보수를 크게 해치는 최적화는 ADR로 사유를 남깁니다.
