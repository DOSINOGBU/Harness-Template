# Data

데이터 변경은 코드 변경보다 되돌리기 어렵습니다.
AI 에이전트가 데이터에 직접 영향을 주는 작업을 할 때 이 문서를 따릅니다.

## Irreversible Operations

| 작업 | 원칙 |
|---|---|
| 컬럼/테이블 삭제 | 사용처를 모두 제거한 뒤 별도 변경으로 진행 |
| 대량 UPDATE/DELETE | 대상 건수를 먼저 SELECT로 확인 |
| 스키마 마이그레이션 | 롤백 스크립트 또는 전략을 함께 준비 |
| 외부 시스템 호출 | 멱등성 확인 후 실행, 실패 시 재시도 영향 기록 |
| 사용자 데이터 삭제 | 사용자 승인 필요 |

## Migration Rules

- 마이그레이션은 한 번에 하나의 의도만 담습니다.
- 새 컬럼은 nullable 또는 default 값으로 추가하고, 코드 배포 후 NOT NULL을 설정합니다.
- 이름 변경은 새 컬럼 추가 → 양쪽 쓰기 → 읽기 이전 → 옛 컬럼 제거의 단계로 진행합니다.
- 마이그레이션 파일은 시간순으로 누적되며 과거 파일은 수정하지 않습니다.

## Backup Before Destructive Change

| 대상 | 권장 절차 |
|---|---|
| 로컬 DB | 더미 데이터를 다시 만들 수 있다면 생략 |
| 공유/스테이징 DB | 변경 전 스냅샷 또는 dump |
| 운영 DB | 사용자 승인과 백업 확인 필수 |
| 운영 파일 시스템 | 변경 전 압축 백업 또는 스냅샷 |

## Verification After Change

- 영향받는 행 수를 사전 추정과 비교합니다.
- 대표 샘플 레코드를 직접 확인합니다.
- 관련 기능의 정상 흐름을 한 번 더 실행합니다.
- 비정상 흐름(빈 결과, 누락 필드)도 함께 확인합니다.
