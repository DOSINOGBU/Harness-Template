# Code Style

도구가 잡지 못하는 판단 기준을 적습니다.
린트와 포매터로 잡히는 규칙은 도구에 위임합니다.

## Naming

- 이름은 역할을 드러내고 약어를 피합니다.
- 같은 개념에 두 단어를 섞지 않습니다(`Glossary` 우선).
- Boolean은 `is`, `has`, `can`처럼 질문 형태로 둡니다.
- 함수는 동사로, 값은 명사로 둡니다.
- `data`, `temp`, `value`처럼 맥락 없는 이름은 피하고 도메인 의미를 드러냅니다.

## Comments

- 기본은 주석을 쓰지 않습니다. 이름이 충분하면 그대로 둡니다.
- 주석을 쓸 때는 "왜"를 적습니다. 코드가 말해주는 "무엇"은 적지 않습니다.
- 외부 사양, 알려진 함정, 우회 이유처럼 코드만 봐서는 알 수 없는 정보를 남깁니다.
- 변경 이력은 커밋 메시지에 적고, 코드에 남기지 않습니다.

## Errors

- 실패를 빈 `catch`로 삼키지 않습니다.
- 사용자에게 보일 메시지와 내부 디버깅 정보를 분리합니다.
- 도메인 의미가 있는 에러는 일반 예외로 감싸지 않고 분류합니다.
- 재시도, 무시, 중단의 정책을 호출 측이 알 수 있게 만듭니다.
- 에러를 다시 던질 때는 원본 의미를 보존하고 입력, 식별자, 단계 같은 맥락을 추가합니다.

## Functions

- 한 함수는 한 가지 일을 합니다.
- 인자 수가 많아지면 객체로 묶거나 책임을 나눕니다.
- 부수효과가 있는 함수와 순수 계산 함수를 섞지 않습니다.
- 깊은 중첩보다 조기 반환을 선호합니다.
- 입력과 출력이 빠르게 예측되지 않는 함수는 더 작은 단위로 나눕니다.

## Responsibility Boundaries

- 데이터 조회, 변환, 비즈니스 판단, UI 렌더링을 한 함수에 섞지 않습니다.
- 입력 검증은 경계에서 먼저 수행하고, 내부 로직은 검증된 값을 받도록 둡니다.
- 상태 변경은 호출 흐름에서 드러나게 하고 숨은 부수효과를 피합니다.

## Files

- 한 파일에 책임이 다른 모듈을 섞지 않습니다.
- 공개 인터페이스는 위에, 내부 도우미는 아래에 둡니다.
- 큰 파일은 도메인 단위로 쪼갭니다.

## File Size Budget

파일 크기는 설계 드리프트를 빨리 발견하기 위한 신호로 사용합니다. 줄 수를 이유로 자동 분리하지는 않지만, 기준을 넘으면 새 기능을 계속 쌓기 전에 책임 분리를 검토합니다.

| 기준 | 처리 |
|---|---|
| 300 lines 이상 | 새 코드 추가 전에 책임 분리 가능성을 검토합니다. |
| 500 lines 이상 | warning으로 보고하고 분리 후보로 기록합니다. |
| 800 lines 이상 | 새 기능 추가를 중단하고 refactor/분리 계획을 먼저 세웁니다. |
| 1200 lines 이상 | failure로 취급합니다. 예외 승인 없이는 PR/커밋하지 않습니다. |

예외 대상:

- generated file
- lockfile
- vendored file
- 빌드 산출물, coverage, dist, node_modules

## CodeHealth Warning Triage

CodeHealth warning은 즉시 수정 지시가 아니라 책임 경계가 흐려졌는지 확인하는 신호로 사용합니다. warning을 없애기 위해 의미 없는 파일 쪼개기나 helper 추출을 하지 않습니다.

- `code-health-large-file` warning은 먼저 파일이 여러 책임을 섞고 있는지 확인합니다.
- 분리가 필요하면 다음 단위를 우선 후보로 검토합니다: dispatch, source helpers, request builder, parser, validator.
- 분리 후 공개 인터페이스가 늘어나면 호출 흐름과 에러 맥락이 더 읽기 쉬워지는지 함께 확인합니다.
- 테스트 fixture, table-driven case, 반복 setup처럼 의도적 반복이 읽기 쉬움과 실패 위치를 보존하면 warning을 수용할 수 있습니다.
