# Project Rules

AI 에이전트와 사람이 함께 지켜야 하는 프로젝트 규칙입니다.
부드러운 권장보다 실행 가능한 제약으로 작성합니다.

## Decision Priority

판단이 충돌하면 아래 순서로 결정합니다.

1. 디버깅 가능성
2. 읽기 쉬움
3. 안정성
4. 성능
5. 짧음

## Constraints

| 제약 | 이유 | 예외 처리 |
|---|---|---|
| 새 의존성은 승인 없이 추가하지 않습니다. | 보안, 번들 크기, 유지보수 영향 | 필요성, 대안, 영향 범위를 먼저 설명 |
| 자동 생성 파일은 직접 수정하지 않습니다. | 재생성 시 변경이 사라짐 | 생성 원본 또는 스키마를 수정 |
| 관련 없는 리팩터링을 섞지 않습니다. | 리뷰와 롤백이 어려움 | 별도 작업 계획으로 분리 |
| 커밋은 검증 가능한 최소 단위로 분리합니다. | 추적, 리뷰, 되돌리기가 쉬워짐 | 커밋 기준은 `docs/VERSION_CONTROL.md`를 따름 |
| 공유 기본 브랜치는 PR을 통해 변경합니다. | 리뷰와 검증 기록을 남기기 위함 | PR 기준은 `docs/PULL_REQUESTS.md`를 따름 |
| 검증 없이 완료를 선언하지 않습니다. | 실제 동작을 확인하지 못하면 회귀를 숨길 수 있음 | 실행 불가 사유와 대체 확인 방법을 기록 |
| 실패를 조용히 무시하지 않습니다. | 원인 추적과 복구가 어려워짐 | 예상 가능한 실패도 처리 이유와 맥락을 남김 |
| 비밀값을 커밋하지 않습니다. | 보안 사고 방지 | `.env.example`에는 값이 아니라 변수명과 설명만 둡니다 |

## Event To Action

| 이벤트 | 필수 행동 |
|---|---|
| 기능 추가 | 요구사항 확인, 관련 문서 확인, 상태 처리와 검증 방법 기록 |
| 버그 수정 | 재현 조건, 가능한 원인, 확인된 원인, 최소 수정, 재검증 기록 |
| 리팩터링 | 동작 변경 없음 확인, 관련 테스트 실행 |
| 설정 변경 | 영향 범위와 롤백 방법 기록 |
| 의존성 추가 | 이유, 대안, 라이선스/보안 영향 기록 |

## Done Criteria

- 요구사항을 충족합니다.
- 관련 테스트 또는 수동 검증을 완료합니다.
- 실패 가능성과 남은 리스크를 기록합니다.
- 새로 알게 된 규칙은 문서나 체크리스트에 반영합니다.
