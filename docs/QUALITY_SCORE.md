# Quality Score

AI 에이전트 작업 결과를 평가하기 위한 기준입니다.
점수 자체보다, 어디가 부족한지 빠르게 찾는 것이 목적입니다.

## Decision Priority

품질 판단은 디버깅 가능성, 읽기 쉬움, 안정성, 성능, 짧음 순서로 봅니다.
짧지만 실패 원인을 찾기 어려운 변경은 좋은 결과로 보지 않습니다.

## Scorecard

| 항목 | 0점 | 1점 | 2점 |
|---|---|---|---|
| 요구사항 충족 | 핵심 요구 누락 | 일부 충족 | 명확히 충족 |
| 변경 범위 | 관련 없는 변경 많음 | 일부 불필요한 변경 | 필요한 범위만 변경 |
| 검증 | 검증 없음 | 제한적 검증 | 관련 검증 완료 |
| 디버깅 가능성 | 실패 원인 추적 어려움 | 일부 로그/에러 있음 | 실패 지점이 명확함 |
| 유지보수성 | 구조를 해침 | 애매한 구조 | 기존 구조에 맞음 |
| 핵심 산출물 품질 | 구조만 있고 실제 사용 가치 없음 | 일부 사용 가능하지만 승인/검수 필요 | 대표 샘플 기준으로 사람이 사용할 수 있음 |
| 의존 기능 준비도 | downstream 기능이 신뢰할 입력이 없음 | 입력은 있으나 품질 기준 또는 승인 상태가 불명확함 | downstream 기능이 신뢰할 입력을 가지고 있음 |

## Minimum Passing Standard

- 핵심 요구사항 충족: 2점
- 검증: 최소 1점
- 디버깅 가능성: 최소 1점
- 관련 없는 변경: 없어야 함
- 핵심 산출물이 downstream 기능의 입력이면 핵심 산출물 품질: 2점
- downstream 기능을 완료로 보려면 의존 기능 준비도: 2점

## Usage

- PR 작성 시 `.github/PULL_REQUEST_TEMPLATE.md`의 Quality Score 체크를 채웁니다.
- 리뷰 요청 시 `.harness/prompts/review-change.md`가 이 기준을 함께 확인합니다.
- PR 초안 작성 시 `.harness/prompts/prepare-pr.md`가 검증, 리스크, 품질 점수 항목을 포함합니다.
- 핵심 산출물 품질 또는 의존 기능 준비도가 2점 미만이면 downstream 품질 완료가 아니라 repair 대상으로 기록합니다.
